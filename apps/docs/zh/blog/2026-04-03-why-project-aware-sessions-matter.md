# 2026-04-03 · 为什么项目感知会话比再多一个 AI 功能更重要

发布时间：2026-04-03  
标签：`产品` `项目上下文` `AI 助手`

作为一个天天在做 AI 产品的人，我现在越来越确定一件事：这个行业最容易犯的错，不是功能做少了，而是又多做了一个“看起来很酷”的功能，却没有把真正决定产品是否可用的那层东西打通。

项目感知会话，就是这种东西。

它听起来一点都不炸裂。它不像接了一个新模型，也不像多了一个会在演示里发光的按钮。可问题在于，如果一个 AI 助手不能真正稳定地贴着你当前的项目工作，那它本质上就不是在帮你工作，它只是在你的工作旁边说话。

### 现在很多 AI 产品，本质上还是“游客模式”

很多 AI 工具今天的状态，说白了还是游客模式。

它能看一眼环境。它能猜到一些事情。它甚至可以在 demo 里猜得很准，于是看起来好像已经很聪明了。

但一旦你要求连续性，问题马上就出来了。

你新建一个会话，选了项目目录，自然会以为接下来这个助手已经知道这个项目自己的规则、项目自己的 skills、项目自己的边界，以及这次工作真正发生在哪里。结果你很快会发现，可能只是 cwd 变了，真正重要的那几条链路根本没跟上。skill 没加载完整，项目自己的 AGENTS 没进上下文，同名 skill 还可能混在一起。最糟糕的地方在于，这种问题不是“完全不可用”，而是“有时候像能用”，于是它会持续消耗你的信任。

这件事为什么重要？因为 AI 产品的信任感，从来不是靠偶尔答得特别好建立的，而是靠上下文是否稳定、是否可预测建立的。

### 半加载的上下文，比没有上下文更糟糕

真正麻烦的不是“没有项目上下文”，而是“看起来有，实际上没跟全”。

如果产品明确告诉你，“我现在还不知道这个项目”，那用户反而容易理解，也容易绕过去。但如果产品表面上已经接受了你的项目选择，实际上却静默漏掉了项目 skill、项目规则或者项目 metadata，这种体验会更坏。因为你不知道问题到底出在哪一层，你只会觉得整个系统不可信。

所以在我看来，项目感知会话根本不是一个便利功能，它是产品完整性的一部分。

当用户选择一个项目时，这个选择就应该贯穿整条会话链路：

- 可用 skills
- 项目自己的规则与说明
- runtime prompt 的构造
- UI 里修改和移除项目的交互入口
- 同名 skill 冲突时的稳定标识

只要这几层里有一两层没跟上，你就没有真的解决问题，你只是把混乱换了个地方放。

### 这里真正重要的，不只是 skill

你当然可以把这次事情描述成“我们把项目 skill 加载修好了”，但我觉得这还是说轻了。

更深一层的问题是：AI 产品的交互，不应该围绕零散控件组织，而应该围绕真实工作上下文组织。

一个项目选择器本身没有价值。一个 header 上的项目 tag 本身也没有价值。一个 session metadata 字段本身也没有价值。只有当这些东西共同指向同一个真实语义，而且整套系统都承认这件事时，它们才真正有价值。

很多 AI 产品就是死在这里。先加一个 UI 控件，再加一个后端字段，再补一个 helper，再补一个 fallback，再补一个兼容分支。最后表面上看已经“支持了”，实际上系统里根本没有一条清晰的端到端契约。

而 AI 产品特别容易掉进这个坑，因为大家总觉得模型够聪明，就能把结构上的问题糊过去。

但这根本行不通。

### 对 NextClaw 来说，这件事其实更核心

NextClaw 的长期目标，从来不是“一个多挂了几个按钮的聊天框”。

它想做的是 AI 时代的个人操作层，是用户通过一个入口去使用软件、服务、互联网和算力的统一界面。如果这是目标，那 session context 就不可能只是一个边角料，它必须是产品最核心的基础设施之一。

这意味着，项目不只是一个路径。它其实是“这次工作应该遵循哪些规则、具备哪些能力、落在哪个边界里”的整体定义。

当用户选择一个项目时，他真正表达的是：

“接下来，请你在这个环境里工作，而不是在这个环境旁边表演。”

这比再多一个命令、再多一个模型开关、再多一个试验性功能都重要。

### 很多真正的产品工作，看起来都很无聊

我做 AI 系统越久，越觉得一个残酷现实是：真正决定产品质量的，往往不是那些最炫的部分。

真正决定质量的，通常是这些看起来有点“无聊”的地方：

- 状态有没有真的传干净
- 同名对象会不会安全地区分
- 新会话是不是行为一致
- 清除上下文是不是和设置上下文一样明确
- UI 展示的是不是系统当前真实状态，而不是过期快照

这类工作几乎不会在演示时赢得欢呼，但它恰恰决定了产品会不会从“有点意思”变成“真的靠谱”。

而“靠谱”，才是能够长期复利的东西。

### 我觉得下一步会越来越清楚

我的判断很简单：AI 助手真正变得更有用，不是因为它们永远再多一个功能，而是因为它们终于不再假装每段对话都从零开始。

未来不只是更强的模型。

未来更重要的是更强的连续性。

会话级上下文、项目级技能、作用域明确的说明、稳定的标识、清晰的控制入口，这些才是让一个 AI 产品看起来“像个认真产品”的基本面。没有这些，功能越加越多，很多时候只是表面积变大了，真实效用并没有一起增长。

所以我会说，项目感知会话比再多一个 AI 功能更重要。

它不是产品最闪亮的部分。

但它是让那些闪亮部分终于值得被信任的部分。
