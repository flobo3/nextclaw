# NextClaw 桌面端更新产品方案（最终版）

## 1. 文档目的

这份文档回答的不是“桌面端能不能更新”这个纯技术问题，而是：

- NextClaw 桌面端的更新，最终应该给用户什么体验
- 哪些事情应该自动做，哪些事情必须由用户明确触发
- 当前已经实现到了哪一层
- 为了把它做成真正可交付的产品，还需要补什么

本文是桌面端更新的**最终产品方案**。  
底层架构仍以已有的 `launcher + versioned bundle + manifest + signature + rollback` 为基础；本次要确定的是**产品行为和默认策略**。

---

## 2. 先给结论

### 最终推荐策略

采用：

**自动检查 + 应用内通知 + 用户可手动下载 + 下载后由用户决定何时应用**

并保留一个可选项：

**允许用户开启“发现更新后自动后台下载”**

### 明确不推荐的默认策略

不推荐默认做成：

- 静默发现更新后自动下载并自动替换
- 没有用户感知地切换到新版本
- 把用户送去网页重新下载 DMG / EXE / AppImage 作为主路径

### 一句话原则

NextClaw 是用户的个人操作层，不是浏览器标签页。  
更新必须追求：

- 应用内闭环
- 用户可见
- 用户可控
- 默认安全
- 尽量少打断

---

## 3. 为什么这样定

NextClaw 和 Chrome 这类产品的核心差异在于：

- Chrome 更偏“基础浏览器容器”，强自动更新的心理预期更强
- NextClaw 更像“个人工作台 + 会话承载层 + 插件承载层 + 本地运行时入口”

这意味着用户对 NextClaw 更新的敏感点更高：

- 会不会打断当前工作
- 会不会改掉当前体验
- 会不会在我不知情的情况下换掉版本
- 会不会影响当前会话、插件、渠道连接、模型配置

因此，NextClaw 不适合把“静默替换”当默认故事。  
更适合参考的是开发工具和协作工具的中间路线：

- 自动发现
- 明确通知
- 应用内一键下载
- 下载后由用户决定何时切换

这条路线既比“完全手动去网页下安装包”更产品化，也比“后台静默替换”更尊重用户。

## 4. 参考产品模式

### 模式 A：VS Code 式更新策略可配置

VS Code 官方文档明确提供 `update.mode`，支持：

- `default`
- `start`
- `manual`
- `none`

这说明顶级产品并不认为“所有用户都该只有一种更新方式”，而是承认更新策略本身就是一个产品设置。

对 NextClaw 的启发：

- 更新行为必须可配置
- 至少要支持自动检查和手动检查
- “完全自动”不应该是唯一选择

参考：
- https://code.visualstudio.com/docs/enterprise/updates

### 模式 B：Slack 式应用内更新闭环

Slack 的 Direct Download 版支持：

- 在应用内检查更新
- 在应用内下载
- 下载后 `Restart to Apply Update`

它不是让用户去网页重新下载作为主路径，而是把更新闭环留在应用里。

对 NextClaw 的启发：

- 你说的“至少支持用户自己点触发下载”，这是对的，而且应该是主能力之一
- “应用内下载 + 用户决定重启”是成熟产品做法

参考：
- https://slack.com/help/articles/360048367814-Update-the-Slack-desktop-app

### 模式 C：JetBrains / Toolbox 式“自动通知 + 用户控制”

JetBrains 的文档说明：

- 默认自动检查并通知
- 可配置更新通道
- 用户可以更新当前实例、忽略、延期
- 某些更新可 patch-based，仅需重启

对 NextClaw 的启发：

- 更新不仅要有“是否有更新”
- 还要有“怎么处理这个更新”
- 应提供：
  - 稍后提醒
  - 忽略此版本
  - 现在应用

参考：
- https://www.jetbrains.com/help/idea/update.html

### 模式 D：Electron 原生 autoUpdater 的边界

Electron 官方文档说明：

- 内建 `autoUpdater` 只原生支持 macOS / Windows
- Linux 没有统一内建更新路径
- macOS 自动更新依赖签名
- `checkForUpdates()` 一旦发现更新，会自动下载

对 NextClaw 的启发：

- 它不适合做你们当前这套跨三平台、无平台签名、统一 bundle 产品单元的主架构
- 但它的“发现后下载、下完后重启应用”交互思想仍可借鉴

参考：
- https://www.electronjs.org/docs/latest/api/auto-updater
- https://www.electronjs.org/docs/latest/tutorial/updates

## 5. 最终产品决策

### 5.1 默认策略

默认使用：

- 自动检查更新：开启
- 自动后台下载：关闭
- 下载后自动应用：关闭

也就是说，默认行为是：

1. 应用启动后或周期性检查更新
2. 如果发现新版本，告诉用户“有新版本可用”
3. 用户点击 `下载更新`
4. 下载完成后，用户点击 `立即重启更新` 或稍后再更新

这是最平衡的默认值。

原因：

- 对用户最可控
- 不会偷偷占带宽
- 不会让用户感觉“版本被背后换掉了”
- 也不会退回到“去网页下载”的旧体验

### 5.2 可选策略

允许在设置中开启：

- 发现更新后自动后台下载

开启后行为变成：

1. 自动检查
2. 自动下载
3. 下载完成后只提示“重启后应用”
4. 不自动重启，不自动切换

这是给更偏效率型用户的配置。

### 5.3 绝不默认做的事

以下行为默认不做：

- 发现后自动下载并立刻替换
- 没有用户感知地自动重启
- 正在使用时偷偷切版本

## 6. 用户可见的更新入口

桌面端最终必须至少有三个入口。

### 入口 A：设置页中的“更新”

放在桌面端设置页中，提供完整状态与操作：

- 当前应用版本（launcher version）
- 当前产品版本（bundle version）
- 当前更新通道
- 上次检查时间
- 当前状态
- 按钮：
  - `检查更新`
  - `下载更新`
  - `立即重启更新`
  - `查看更新说明`

### 入口 B：帮助菜单 / 应用菜单

至少补：

- `Check for Updates`
- `Restart to Apply Update`（仅当已下载时显示）

### 入口 C：非打断式通知入口

建议用以下任一种：

- 顶部 banner
- 设置按钮右上角 badge
- 托盘 / 菜单项 badge

原则是：

- 有更新时可见
- 不强打断
- 用户随时能回来处理

## 7. 状态机

更新体验必须围绕一个显式状态机展开，不能只靠弹窗和隐式日志。

建议状态如下：

- `idle`
  - 当前无更新动作
- `checking`
  - 正在检查更新
- `update-available`
  - 发现新版本，但尚未下载
- `downloading`
  - 正在下载更新
- `downloaded`
  - 已下载完成，等待用户应用
- `updating-on-restart`
  - 已确认将在下一次重启应用
- `up-to-date`
  - 已是最新
- `failed`
  - 检查、下载或校验失败
- `rollback-applied`
  - 新版本失败后已自动回滚

状态机必须可序列化到本地，这样：

- 重启后还能恢复状态
- UI 可以可靠展示“已下载待应用”
- 不会出现用户重开应用后完全不知道发生了什么

## 8. 用户感知设计

### 场景 1：发现新版本但尚未下载

提示文案建议：

- 标题：`发现新版本`
- 文案：`NextClaw 有可用更新。你可以现在下载，也可以稍后处理。`

按钮：

- `下载更新`
- `稍后`
- `查看更新说明`

### 场景 2：下载中

显示：

- 下载进度
- 当前版本 -> 新版本
- 可取消 / 后台继续

### 场景 3：已下载完成

提示文案建议：

- 标题：`更新已准备好`
- 文案：`新版已经下载完成。重启后将应用新的 UI、runtime 和内置插件。`

按钮：

- `立即重启更新`
- `稍后`

可选：

- `退出时自动应用`

### 场景 4：更新失败

提示文案建议：

- 标题：`更新失败`
- 文案：`更新未能完成，当前版本未受影响。你可以稍后重试。`

按钮：

- `重试下载`
- `查看日志`

### 场景 5：回滚

提示文案建议：

- 标题：`已恢复到上一版本`
- 文案：`新版启动失败，NextClaw 已自动恢复到上一可用版本。`

按钮：

- `查看详情`

## 9. 设置项设计

建议新增 `设置 > 更新` 页面，至少有以下项：

### P0 必做

- `自动检查更新`
  - 默认：开
- `检查更新`
  - 按钮
- `下载更新`
  - 仅在 `update-available` 状态显示
- `立即重启更新`
  - 仅在 `downloaded` 状态显示

### P1 建议补

- `发现更新后自动后台下载`
  - 默认：关
- `退出应用时自动应用已下载更新`
  - 默认：关
- `忽略此版本`
- `更新通道`
  - `stable`
  - `beta`

### P2 再考虑

- 指定更新时间窗口
- 静默下载带宽策略
- 企业策略覆盖

## 10. 当前实现与目标方案的差距

### 已经有的

当前代码已经具备以下能力：

- 启动后检查远端 manifest
- 下载新的 product bundle
- 校验 hash / signature
- 激活 candidate
- 重启后切换新 UI / runtime / bundled plugins
- 启动失败自动回滚

### 当前还不符合最终产品方案的地方

1. **下载触发权太偏系统，不够偏用户**
   - 当前后台检查路径会直接进入 `stageUpdate`
   - 这相当于“发现即下载”
   - 不符合本方案的默认策略

2. **缺少显式更新状态 UI**
   - 现在主要靠弹窗和日志
   - 用户看不到完整状态机

3. **缺少应用内手动入口**
   - 需要明确的：
     - `检查更新`
     - `下载更新`
     - `立即重启更新`

4. **缺少更新设置**
   - 用户目前无法控制自动下载策略

5. **缺少更新说明与版本可见性**
   - 至少要知道：
     - 现在是什么版本
     - 新版是什么
     - 这次更新了什么

## 11. 实施建议

### P0：把“能工作”变成“可交付”

这一阶段做完后，就可以说桌面更新体验已经真正产品化。

必须完成：

1. 将后台逻辑拆成两步
   - `checkForUpdate`
   - `downloadUpdate`

2. 修改默认行为
   - 自动检查
   - 发现后进入 `update-available`
   - 默认不自动下载

3. 增加更新状态 owner
   - 统一管理：
     - 状态
     - 当前版本
     - 远端版本
     - 上次检查时间
     - 下载进度
     - 错误信息

4. 增加应用内更新入口
   - 设置页
   - 菜单项

5. 增加“下载完成后应用”交互
   - `立即重启更新`
   - `稍后`

### P1：把“可交付”变成“更像顶级产品”

建议继续补：

1. 自动后台下载设置
2. 忽略此版本
3. Release Notes 展示
4. 托盘 / banner badge
5. 退出时应用已下载更新

### P2：以后再做

1. launcher self-update
2. 企业策略覆盖
3. 灰度 / staged rollout
4. delta update

## 12. 我对“是否已经完成”的最终判断

### 如果目标是：

“发布新版功能后，用户不需要去网页重新下载安装包，就能在应用内拿到新 UI / runtime / 插件版本”

那么：

**底层机制已经基本完成。**

### 如果目标是：

“桌面更新已经是一个成熟、清晰、用户可控、可设置、可理解的产品体验”

那么：

**还没有完成。**

差的不是底层下载/校验/切换能力，而是：

- 默认策略
- 用户控制权
- 状态可见性
- 设置入口

---

## 13. 最终建议

我建议把这件事收口成下面这句产品原则，并按它实现：

> NextClaw 桌面端更新默认自动检查，但不默认静默替换；更新必须在应用内完成发现、下载和应用，并由用户决定何时切换到新版本。

这比“纯自动”更适合 NextClaw，  
也比“手动去网页下安装包”更像顶级产品。
