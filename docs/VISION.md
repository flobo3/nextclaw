# NextClaw 产品愿景

## 一句话愿景

NextClaw 的长期愿景，是成为 AI 时代的个人操作层：它不是传统意义上的操作系统内核，而是用户使用软件、互联网、系统、服务与云计算的默认入口。

## 它到底是什么

NextClaw 想做的，不是一个“功能很多的 AI 产品”，而是一个足够统一、足够自然、足够强大、足够可长期依赖的 AI 原生入口。

理想状态下，用户不需要先想“我要打开哪个软件、哪个网站、哪个服务”，而是先进入 NextClaw，说出目标，由 NextClaw 去理解意图、调用工具、连接服务、组织流程，并把结果交付出来。

如果传统操作系统定义了“人如何使用计算机”，那么 NextClaw 想定义的是“人如何在 AI 时代使用数字世界”。

## 愿景的完整结构

NextClaw 的愿景不该只剩一句“个人操作层”，也不该只剩一组孤立的功能支柱。更准确的写法是：

**“个人操作层”是上位目标；“统一入口、能力编排、自感知与自治、自进化、生态扩展、数字基础设施、开箱即用”是把这个目标真正做成产品所必须同时成立的几条主线。**

### 1. 统一入口

NextClaw 首先要成为统一入口，而不是又一个平行工具。

- 用户进入数字世界时，应该越来越倾向于先打开 NextClaw
- 软件、服务、渠道、设备与云资源，应该尽量通过 NextClaw 被统一接入与管理
- 它应该逐步成为能承接工作流、沟通流和信息流的默认工作台

这也是“个人操作层”最外层、最容易被用户直接感知到的形态。

### 2. 能力编排

NextClaw 的价值不在于把所有功能都自己重做一遍，而在于更好地连接、调度、组合现有的软件、系统、服务与算力。

- 把用户意图翻译成可执行流程
- 调用工具、连接服务、组织多步任务
- 在多模型、多渠道、多插件、多运行环境之间收敛成统一体验

我们追求的是“统一入口与能力编排”，不是“功能垃圾场”。

### 3. 自感知与自治

原来的“自知自治”方向不应该被丢掉，但术语上没必要拆成两个近义词。更准确的表达应该是：**用“自感知”统一承接原本“自知”的含义，再与“自治”配对。**

#### 自感知

NextClaw 不能只是一个收到输入再吐出回答的壳。它应该逐步具备产品意义上的自感知能力，也就是：

- 知道自己是谁：身份、版本、规则、能力边界、工具边界是什么
- 知道自己当前的系统状态：完整配置、运行状态、健康度、服务连接状态是什么
- 知道自己在为谁工作：用户是谁、偏好是什么、历史习惯是什么
- 知道自己处在什么场景：当前会话、项目、渠道、设备、运行环境是什么
- 知道自己做过什么：刚执行了哪些动作、读过哪些上下文、调用过哪些资源
- 知道结果如何：哪些任务成功了、失败了、卡住了、偏离了
- 知道哪里做错了：问题来自模型判断、工具调用、上下文缺失，还是执行链路本身

这里说的不是抽象哲学意义上的“意识”，而是产品化、系统化、可落地的情境感知能力。没有这层能力，系统就很难真正成为长期工作台，只会停留在“每轮都像重新开始”的聊天工具。

#### 自治

在具备足够自感知之后，NextClaw 还应该逐步具备自治能力：

- 能通过工具执行系统中的关键管理动作，例如修改配置、启停渠道、管理定时任务、安装或卸载扩展、触发诊断与恢复
- 能在聊天、UI 与系统控制面之间形成一等入口，而不是让“能说话”和“能管理系统”长期割裂
- 对高风险或不可逆操作保留明确确认机制，确保安全可控

目标不是“神化 agent”，而是让用户通过自然语言与统一控制面，就能理解和掌控 NextClaw 自身。

### 4. 自进化

NextClaw 也不能是一个静态入口。它应该具备可治理、可积累、可复用的自进化能力，也就是：

- 能从错误中学习，而不是反复犯同样的问题
- 能从用户反馈、任务结果、失败回放里做反思和修正
- 能逐步沉淀更好的执行策略、工具使用纪律、上下文组织方式和记忆结构
- 能把一次次经验沉淀为更长期的用户偏好、项目知识、procedural memory 与 skill memory
- 能随着使用时间增长，越来越像“这个用户自己的操作层”，而不是永远停留在通用默认形态

这里说的不是不可控的自动自我改写。我们要的是可观察、可约束、可回退、可治理的自我改进能力。

### 5. 插件化拓展与生态繁荣

原来的“插件化拓展与生态繁荣”同样应该保留，因为个人操作层不能靠内核自己无限膨胀，而要靠稳定内核加扩展生态来成长。

- 核心运行时、插件 SDK、渠道协议需要逐步收敛并保持稳定边界
- 复杂能力的长期扩张，应更多通过 plugin、skill、marketplace 来完成
- 用户与开发者应该能形成“使用 -> 开发 -> 分享 -> 发现”的闭环

这条主线的意义，是让 NextClaw 追求“能力可无限扩展”，但又不把所有复杂度都压回主产品本体。

### 6. 数字世界基础设施

NextClaw 不只是一个聊天机器人，它致力于成为用户在数字世界中的通用操作基础设施，主要体现在三个维度：

#### 宿主系统的理解与利用

- 理解和利用操作系统、文件系统、本地服务、硬件资源等宿主环境
- 通过工具调用完成文件管理、进程管理、系统监控、本地应用联动等操作
- 让 AI 成为用户操控本地环境的智能中枢

#### 互联网数据与云计算能力的利用

- 便捷接入搜索、抓取、API 调用、信息聚合等互联网数据源
- 调度和利用云存储、云函数、远程服务等云资源
- 让用户通过自然语言即可触达互联网世界的数据与算力

#### 用户数据的统一管理

- 管理用户的本地数据、配置、笔记与个人知识
- 连接散落在邮件、日历、社交、笔记、云盘等各类系统中的数据
- 在尊重隐私与安全的前提下，提供统一的数据视图与操作能力

### 7. 开箱即用

“个人操作层”不能只对高阶用户成立，也必须对新用户成立。所以“开箱即用”不是一个可有可无的体验目标，而是愿景落地的必要条件。

- 首次启动就应让用户知道它能做什么
- 关键场景应支持一键配置与启动，而不是先学习大段配置
- 从了解、安装到第一次真正跑通，应尽可能短、尽可能顺滑

如果一个产品理论上很强，却很难真正进入用户日常，它就很难成为默认入口。

## 我们追求的产品级别

我们追求的不是复刻电影里的 JARVIS 设定，而是接近那种产品级别的存在：

- 足够重要，能成为用户的第一入口
- 足够统一，能承接软件、服务、平台与算力
- 足够自然，让用户更多表达意图，少做工具切换
- 足够连续，不会每一轮都像失忆
- 足够可靠，让人愿意长期把它当成默认工作台
- 足够会成长，越用越贴近用户而不是长期停在原地

换句话说，NextClaw 想成为的不是“又一个工具”，而是未来用户最先打开、最常依赖、最能代表其数字能力边界的那个平台。

## 这不意味着什么

这份愿景不是说：

- 要把所有能力都硬塞进一个单体产品
- 要把一切都自己重做一遍
- 要为了看起来全能而持续堆功能
- 要做一个只有聊天外壳、却不能真正调度世界的 AI
- 要把“自感知”写成玄学叙事，却没有真实系统能力支撑
- 要把“自进化”做成不可控的自动自我修改黑盒

如果一个能力不能增强 NextClaw 作为统一入口的地位，不能增强它对用户、场景、历史与结果的感知，不能增强它从反馈与错误中持续进化的能力，不能增强它对数字世界的真实调度与接入能力，而只是在增加孤立功能点，那么它的优先级应该下降。

## 愿景约束下的产品与架构判断原则

后续做产品、架构、交互、命名和路线判断时，默认都要回到下面几条原则：

1. **入口优先**
   新能力应该让用户更愿意先打开 NextClaw，而不是让用户先去别的工具，再把结果带回来。
2. **编排优先**
   优先连接、调度、组合现有软件、服务、系统与云资源，而不是无边界地把所有实现都内置进去。
3. **意图优先**
   产品应该越来越擅长把“用户想完成什么”转成“系统具体怎么执行”，而不是把复杂的工具细节重新丢回给用户。
4. **自感知优先**
   系统应该越来越清楚“自己是谁、在什么场景、做了什么、结果如何、错在哪里”，而不是每轮都靠临时 prompt 硬撑连续性。
5. **自治优先**
   用户应该越来越能通过自然语言和统一控制面理解并掌控系统，而不是长期依赖分裂的 UI/CLI/手工配置路径。
6. **自进化优先**
   系统应该越来越能从错误、反馈和长期使用中学习，把经验沉淀成更稳定的能力，而不是把每次失败都当成一次性事故。
7. **生态优先**
   长期扩展能力应优先通过稳定内核和扩展生态承接，而不是让主产品持续膨胀。
8. **统一体验优先**
   不同模型、渠道、插件、服务和运行环境，最终都应尽可能收敛到一致、可理解、可管理的体验。
9. **长期地位优先**
   我们追求的不只是多一个功能点，而是逐步建立一种长期地位：以后用户主要通过 NextClaw 来使用数字世界。

## 与 NCP 的关系

NextClaw 的产品愿景，和 NCP 的基础设施愿景不是一回事，但二者强相关：

- `NextClaw` 是面向用户的产品化操作层
- `NCP` 是支撑这个操作层的通用协议、运行时与积木体系

可以把 NCP 理解为基础设施底座，把 NextClaw 理解为面向真实用户的产品化入口。

因此，NCP 的设计要服务“可组合、可扩展、可复用”的基础设施目标；NextClaw 的设计要服务“统一入口、统一体验、统一能力编排、自感知连续性、自治控制能力、自进化能力积累”的产品目标。

## 对 AI 协作者的默认要求

任何 AI 在这个仓库里做方案、写文档、改代码时，默认都应把这份愿景当成上位目标，并先回答这些问题：

- 这项改动是在增强 NextClaw 作为统一入口的地位，还是只是在堆一个局部功能？
- 这项改动是在提升意图到执行的闭环能力，还是让用户自己承担更多工具复杂度？
- 这项改动是在增强系统对用户、项目、环境、历史、结果与错误的感知，还是继续让系统处于“低上下文、低连续性”的状态？
- 这项改动是在增强用户对系统自身的理解与掌控，还是让系统管理能力继续分裂在多个入口里？
- 这项改动是在帮助系统从反馈与错误中持续学习，还是让同类问题只能重复人工兜底？
- 这项改动是在强化统一体验，还是继续制造碎片化入口？
- 这项改动是在增强编排与接入能力，还是把产品推向“什么都往里塞”的方向？

如果答案偏向后者，就应该重新审视方案。
